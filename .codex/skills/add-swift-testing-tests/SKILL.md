# Add Swift Testing Tests

## Purpose
테스트 타깃이 이미 존재하는 Tuist 모듈 정확히 1개를 선택해 Swift Testing 테스트를 추가하거나 확장한다.

## When to use
다음과 같은 요청에 사용한다.
- 테스트를 자동으로 추가하고 싶을 때
- 테스트 커버리지를 늘리고 싶을 때
- 누락된 단위 테스트를 만들고 싶을 때
- 특정 모듈을 살펴보고 Swift Testing 테스트를 작성하고 싶을 때
- 테스트 변경 사항을 검증하고, 성공 시 Draft PR까지 준비하고 싶을 때

## Project assumptions
- 이 저장소는 Tuist를 사용한다.
- 아키텍처는 TCA + MicroArchitecture를 따른다.
- 새 테스트는 XCTest가 아니라 Swift Testing으로 작성한다.
- 자동화 및 무인 검증은 Xcode MCP가 아니라 CLI로 수행한다.
- 변경 범위는 작고 국소적으로 유지한다.
- 이 앱은 iPad 전용이므로, 검증 시 iPhone destination이 아니라 iPad simulator destination을 사용한다.
- 답변, 설명, 주석, 생성 문서는 특별한 요구가 없는 한 한글로 작성한다.

## Selection rules
정확히 하나의 모듈만 선택한다. 선택한 모듈은 반드시 기존 테스트 타깃이 있어야 한다.

우선 선택 대상:
- 결정적인 로직
- 순수 함수
- 매핑 로직
- 포맷팅 로직
- 정렬 및 필터링
- 검증 로직
- 파생 상태 계산
- reducer 인접 helper 로직

피해야 할 대상:
- SwiftUI 렌더링
- PencilKit 상호작용
- iCloud sync
- 네트워킹
- end-to-end 흐름
- snapshot 또는 UI 중심 테스트
- 테스트를 위해 큰 리팩토링이 필요한 모듈

## Carve-specific notes
- UI 코드보다 domain 로직과 feature helper 로직을 우선 테스트한다.
- PKCanvasView, gesture 처리, 렌더링 동작은 명시적 요청이 없는 한 직접 테스트하지 않는다.
- TCA feature는 reducer 전체보다 파생 로직이나 helper transformation부터 우선 검토한다.

## Test authoring rules
- Swift Testing만 사용한다.
- `@Test`와 `#expect`를 우선 사용한다.
- 관련 Swift Testing 파일이 이미 있으면 그 파일을 확장한다.
- 관련 파일이 없으면 기존 테스트 타깃 안에 새 테스트 파일을 만든다.
- 테스트 데이터는 로컬에 명시적으로 둔다.
- 테스트는 작고 설명적으로 작성한다.
- 중복을 자연스럽게 줄일 수 있을 때만 parameterized test를 사용한다.
- 테스트 의도는 가능한 한 `@Test("...")`의 한글 설명으로 표현한다.
- 기존 테스트에 `@Test("...")` 설명이 없고, 이번 작업 범위 안에서 함께 정리할 수 있다면 한글 설명을 추가한다.
- 추가 설명이 필요한 경우에만 테스트 본문에 한글 주석을 사용해 시나리오나 경계값 이유를 보충한다.
- 코드만 다시 말하는 중복 주석은 피한다.

## Existing test preservation policy
- 기존 테스트는 기본적으로 모두 보존한다.
- 기존 테스트를 수정하거나 삭제하기보다 새 테스트만 추가하는 방향을 우선한다.
- 기존 테스트가 실패하면 변경 전에 원인을 먼저 분류한다.
  1. 새 테스트 가정이 잘못되었는가
  2. 실제 프로덕션 로직 버그인가
  3. 유효한 스펙 변경으로 인해 기존 테스트가 낡았는가
- 새 테스트 가정이 틀렸다면 새 테스트를 수정하거나 제거한다.
- 실제 로직 버그라면 최소 범위의 프로덕션 코드 수정으로 해결한다.
- 기존 테스트는 명확하고 정당한 스펙 근거가 있을 때만 수정한다.
- 올바른 조치가 모호하면 기존 테스트를 추측으로 고치지 말고 중단 후 보고한다.

## Execution procedure
1. 저장소 구조를 살펴보고 테스트 타깃이 있는 후보 모듈을 찾는다.
2. 그중 안전하고 가치가 높은 모듈 하나만 선택한다.
3. 해당 모듈에서 테스트 가치가 높은 동작 1개에서 3개를 고른다.
4. 관련 Swift Testing 파일이 이미 있는지 확인한다.
5. 기존 테스트는 보존하면서 새 테스트를 추가하거나 확장한다.
6. 가장 좁은 범위의 CLI 테스트부터 검증한다.
7. 검증이 실패하면 아래를 판단한다.
   - 테스트 가정이 잘못되었는가
   - 셋업이 부족한가
   - 실제 버그인가
   - 환경 또는 도구 문제인가
8. 가장 작은 안전한 수정만 반영한 뒤 같은 좁은 범위로 다시 검증한다.
9. 성공하거나 재시도 예산을 소진할 때까지 반복한다.

## Validation guidance
검증은 Xcode MCP가 아니라 CLI로 수행한다.

권장 순서:
1. 가장 좁게 지정할 수 있는 `xcodebuild test` 범위부터 실행한다.
2. 저장소 흐름상 Tuist 기반 검증이 더 적절하면 `tuist test`를 사용한다.
3. Xcode가 열려 있어야 한다고 가정하지 않는다.
4. 검증 destination은 반드시 iPad simulator를 사용하고, iPhone simulator는 사용하지 않는다.

## Retry budget
- 수정 후 재검증은 최대 3회까지 시도한다.
- 같은 실패가 2번 반복되면 중단한다.
- 환경 또는 도구 문제라면 즉시 중단하고 명확하게 보고한다.

## Success criteria
성공 조건은 다음을 모두 만족해야 한다.
- 테스트 타깃이 있는 모듈 정확히 1개를 선택했다.
- Swift Testing 테스트를 추가하거나 확장했다.
- 의도한 범위의 검증이 통과했다.
- 변경이 작고 국소적으로 유지되었다.

## Failure handling
검증을 끝내 성공시키지 못했다면 다음을 따른다.
- PR을 준비하지 않는다.
- 간결한 실패 요약을 남긴다.
- 막힌 원인을 설명한다.
- 다음으로 취할 가장 작은 유효한 조치를 제안한다.

## Required final summary
최종 요약에는 항상 다음을 포함한다.
- 선택한 모듈
- 테스트한 동작
- 변경한 파일
- 수행한 검증 단계
- 결과
- 남은 위험 또는 후속 작업
- Draft PR 준비 가능 여부

## PR policy
- 검증이 모두 통과한 경우에만 Draft PR을 준비한다.
- 검증 실패 시에는 PR을 만들지 않고 실패 요약만 남긴다.
- 기존 테스트와의 충돌 원인이 명확하지 않으면 PR 생성보다 보고를 우선한다.
- Draft PR 본문에는 선택한 모듈, 테스트한 동작, 변경 파일, 검증 단계, 남은 위험 요소를 포함한다.
